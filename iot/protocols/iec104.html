IEC104（IEC 60870-5-104）

ACK 机制

ACK 的定义（核心）：ACK = 对端发送的 Nr（Receive Sequence Number）

Nr = 对端已正确接收的 I帧数量

  • IEC104 没有独立 ACK 报文
  • ACK 通过以下两种方式携带：
类型是否ACK
I帧（信息帧）✅（Nr字段）
S帧（监督帧）✅
U帧（测试/控制）❌

总召唤（C_IC_NA_1）流程

客户端 → I帧：ACTIVATION（总召唤）

服务端 → I帧：ACTIVATION_CON
服务端 → 多个 I帧：点表数据（带 Nr）
服务端 → I帧：ACTIVATION_TERMINATION

⚠️ 关键点

  • ACTIVATION_CON ❌ 不是 ACK
  • ACTIVATION_TERMINATION ❌ 不是 ACK
  • 只有 I帧中的 Nr 才是 ACK

t1 超时（断连根因）

客户端发送 I帧后，长时间未收到对端 ACK（Nr未推进） 结果：触发 t1 → 主动断开连接

测试帧（TESTFR）机制

  1. 帧类型

    类型说明
    TESTFR_ACT测试请求（U帧）
    TESTFR_CON测试响应（U帧）

  2. 触发条件（t3）

    链路空闲超过 t3 → 发送 TESTFR_ACT

    默认：≈20s（实现相关）

  3. 交互流程

    A → TESTFR_ACT
B → TESTFR_CON

  4. 超时机制

    发送 TESTFR_ACT 后等待 TESTFR_CON
超时（t1）→ 断连

  5. 与 ACK 的关系

    项目TESTFR
    是否带序号❌
    是否更新 Nr❌
    是否作为 ACK❌

常见误区

误区1：认为“有返回数据就等于ACK”

✔ 错，必须看： Nr 是否推进

误区2：认为“总召唤响应就是ACK”

✔ 错：ACTIVATION_CON / TERMINATION ≠ ACK

误区3：用 testCommand 当心跳

C_TS_NA_1 = I帧（业务帧）≠ TESTFR ✔ 正确心跳： TESTFR_ACT / TESTFR_CON（U帧）

误区4： 认为 TESTFR 能避免 t1 断连

TESTFR 只保活链路，不参与确认机制

✔ I帧

 → handleReceiveSequenceNumber()

✔ S帧

case S_FORMAT:
 → handleReceiveSequenceNumber()

T1超时检查启动条件

send(ASdu)

// 触发
timeoutManager.addTimerTask(maxTimeNoAckReceived)

T1超时检查 取消条件

handleReceiveSequenceNumber() {
    ...
    maxTimeNoAckReceived.cancel();
    
    // 重新启动 T1 的关键条件
    if (sendSequenceNumber != acknowledgedSendSequenceNumber) {
    	timeoutManager.addTimerTask(maxTimeNoAckReceived);
	}
}

断连直接原因

MaxTimeNoAckReceivedTimer.execute()
 → close()

关键状态变量

变量含义
sendSequenceNumber已发送I帧序号
acknowledgedSendSequenceNumber已确认I帧序号
receiveSequenceNumber已接收对端I帧序号
